# 企业端面试会话心跳与增量同步设计

> 设计日期：2026-05-08
> 适用范围：企业端 ChatPage、useInterview、InterviewStorage、/api/interview-sessions

## 1. 目标

为企业端面试增加后端心跳与增量同步能力，解决以下问题：

- 浏览器崩溃或刷新后，仅依赖 localStorage 恢复不够稳妥。
- 当前 conversation 主要保存在前端，无法实时在服务端保留中间进度。
- 企业端需要更可靠地恢复未完成面试，并为后续审计或诊断提供真实会话快照。

目标原则：

- 不改变当前 useInterview 作为主状态机的定位。
- 不要求每个 token 或每次 chunk 都同步。
- 优先采用“答完一题即同步 + 周期心跳”的低侵入模式。

## 2. 当前基础能力

现有后端已经具备：

- GET /interview-sessions/current
- GET /interview-sessions
- POST /interview-sessions
- PATCH /interview-sessions/:id

现有存储表 interview_sessions 已包含：

- status
- conversation
- scoring
- metadata
- start_time
- end_time
- updated_at

因此当前不需要先加新表，即可先完成第一版心跳与增量同步。

## 3. 当前缺口

虽然已有 PATCH 接口，但当前企业端主链路没有把答题进度持续同步到后端。

直接后果：

- 第 7 题中断时，后端可能完全不知道当前题目数和已答内容。
- ErrorBoundary 恢复和刷新恢复主要依赖 localStorage，而不是服务端快照。
- 无法准确判断会话是否“僵死”。

## 4. 同步策略

建议采用双轨同步：

### 4.1 增量同步

触发时机：

- 开始面试后首题生成完成
- 每次候选人提交回答后
- 每次新问题生成完成后
- 评分完成后
- 面试完成或取消时

同步内容：

- status
- conversation.questions
- conversation.candidateAnswers
- metadata.totalQuestions
- metadata.totalAnswers
- metadata.averageAnswerLength
- metadata.lastSyncedQuestionIndex
- metadata.lastSyncedAt

### 4.2 心跳同步

触发时机：

- 面试处于 active 或 scoring 阶段时，每 20 到 30 秒一次

同步内容：

- status
- metadata.lastHeartbeatAt
- metadata.clientPhase
- metadata.currentQuestionIndex

心跳的目的不是传完整 conversation，而是让服务端知道会话仍存活。

## 5. 建议的前端实现方式

### 5.1 useInterview 增加 syncSessionCheckpoint

建议新增一个内部方法：

```text
syncSessionCheckpoint(reason, options)
```

输入：

- reason：例如 question_generated、answer_submitted、heartbeat、completed
- options.includeConversation：是否带完整 conversation
- options.includeScoring：是否带 scoring

行为：

1. 从 InterviewStorage.getCurrentSession() 读取最新会话
2. 组装 PATCH payload
3. 通过 /api/interview-sessions/:id 同步
4. 更新 metadata.lastSyncedAt

### 5.2 心跳计时器

建议由 useInterview 或后续的 useInterviewPersistenceBridge 管理：

- 面试开始后启动 interval
- 面试 completed、cancelled、failed 后清理 interval
- 仅在最近一次同步后超过阈值时发送心跳

## 6. PATCH Payload 建议

建议第一版继续复用现有 upsert 能力，但在 payload 结构上统一：

```json
{
  "id": 123456789,
  "candidateId": 1001,
  "candidateName": "张三",
  "position": "前端开发工程师",
  "status": "in_progress",
  "conversation": {
    "questions": [],
    "candidateAnswers": [],
    "aiReplies": [],
    "timestamps": []
  },
  "metadata": {
    "totalQuestions": 10,
    "totalAnswers": 4,
    "averageAnswerLength": 128,
    "sessionDuration": 12,
    "currentQuestionIndex": 4,
    "lastHeartbeatAt": "2026-05-08T02:00:00.000Z",
    "lastSyncedAt": "2026-05-08T02:00:00.000Z",
    "lastSyncReason": "answer_submitted",
    "clientPhase": "waiting_answer"
  }
}
```

## 7. 冲突与覆盖策略

当前第一版建议采用“最后写入覆盖”，但增加两个保护规则：

- 只有相同 owner_user_id 的会话允许更新。
- completed 或 cancelled 会话默认拒绝被后续 heartbeat 覆盖。

后端建议增加以下判断：

- 如果现有 status 已是 completed，而新请求是 in_progress + heartbeat，则拒绝覆盖。
- 如果 metadata.lastSyncedAt 新于请求时间，则可选择忽略旧请求。

## 8. 服务端增强建议

当前可以不改表结构直接落地，但建议在后续增强：

### 8.1 metadata 标准字段

统一约定 metadata 中包含：

- currentQuestionIndex
- clientPhase
- lastHeartbeatAt
- lastSyncedAt
- lastSyncReason
- version

### 8.2 僵死会话识别

建议后续在 GET /interview-sessions/current 返回时附带：

- isStale：最近心跳已超阈值
- staleMinutes：距离上次心跳的分钟数

这样前端恢复时可以提示：

- 检测到一个长时间未更新的面试会话，是否继续恢复？

## 9. 最小实施顺序

### 阶段 1：前端接现有 PATCH

- useInterview 在首题生成后创建会话
- 每答完一题同步一次完整 conversation
- 面试完成时同步 completed + scoring

完成标准：

- 浏览器刷新后，后端可取回近似完整会话

### 阶段 2：加入心跳

- active/scoring 期间每 20 到 30 秒发送 metadata 心跳

完成标准：

- 后端可判断会话活跃程度

### 阶段 3：恢复策略增强

- 前端恢复提示优先参考后端 current session
- localStorage 作为离线兜底而非唯一来源

完成标准：

- 多设备或跨标签页恢复时，优先使用服务端会话

## 10. 风险与注意事项

- 不要在每次输入字符时同步，会造成噪声和资源浪费。
- 心跳不能覆盖 completed 结果。
- conversation 可能较大，PATCH 时应限制频率。
- 若未来 conversation 继续膨胀，应考虑 questions/answers 的增量 append 接口，而不是每次全量覆盖。

## 11. 建议的后续实施任务

建议后续按以下顺序推进：

1. 在 useInterview 中增加 syncSessionCheckpoint。
2. 打通企业端 /api/interview-sessions/:id 的实际调用。
3. 为 completed 会话增加防覆盖保护。
4. 再补服务端 current session 优先恢复逻辑。
