# 企业端虚拟人统一状态机设计稿

> 设计日期：2026-05-08
> 作用范围：企业端 ChatPage、useInterview、VirtualHumanSDK、VirtualHumanPanel

## 1. 设计目标

基于现有企业端面试主链路，建立一套统一状态机，使以下能力不再分裂：

- 面试问题生成
- 候选人作答提交
- 实时评分
- 虚拟人播报
- 异常降级与恢复

统一后的原则：

- 面试流程只允许一个真状态源，使用 useInterview 作为主状态机。
- VirtualHumanSDK 只负责连接、播报、录音、连接状态上报，不再持有独立的面试流程真相。
- 当虚拟人不可用时，面试流程自动退回文本模式，问题、评分、会话持久化不受影响。

## 2. 当前实现结论

当前代码已经具备以下基础：

- ChatPage 已通过 useInterview 管理企业端面试流程。
- VirtualHumanSDK 已可挂载到页面，并可提供纯播报能力。
- ChatPage 已具备可恢复面试会话和 ErrorBoundary 兜底。

当前仍存在的结构性问题：

- VirtualHumanSDK 内部仍保留 useInterviewState，存在与 useInterview 的双轨状态机。
- 虚拟人连接状态与面试状态仍是并列关系，不是从属关系。
- 候选人语音输入、虚拟人播报完成、自动推进下一题等事件还没有统一进入 useInterview 的事件流。

## 3. 目标架构

目标架构如下：

```text
ChatPage
  ├─ useInterview                // 唯一面试状态机
  ├─ useVirtualHumanBridge       // 虚拟人桥接层
  ├─ Chat UI                     // 文本消息、输入、恢复提示、结果展示
  └─ VirtualHumanPanel
       └─ VirtualHumanSDK        // 只处理连接、播报、录音、状态上报
```

职责边界：

- useInterview
  - 维护面试 phase、问题索引、当前问题、评分、恢复数据
  - 负责问题生成、答案提交、评分与最终报告
- useVirtualHumanBridge
  - 监听 useInterview 的状态变化
  - 决定何时让虚拟人播报、何时请求录音、何时标记降级
  - 将 VirtualHumanSDK 事件转换成 useInterview 可消费的事件
- VirtualHumanSDK
  - 暴露 connect、disconnect、speakText、startRecording、stopRecording
  - 暴露连接状态和播放状态
  - 不再负责题目推进或面试结束逻辑

## 4. 统一状态定义

### 4.1 面试主状态

建议 useInterview 继续作为主状态机，状态收敛为：

- idle：未开始
- preparing：准备面试，会话初始化、生成首题
- asking：问题已生成，等待虚拟人或文本完成展示
- waiting_answer：等待候选人回答
- evaluating：正在做实时评分
- transitioning：准备下一题
- scoring：生成综合报告并保存
- completed：面试完成
- interrupted：会话中断但可恢复
- failed：主链路失败，需要用户重试或降级

### 4.2 虚拟人从属状态

虚拟人状态不再驱动流程，只提供能力状态：

- disconnected：未连接
- connecting：连接中
- connected：可播报、可录音
- degraded：功能可用但不稳定，例如只能播报不能录音
- error：当前连接失败

## 5. 事件模型

建议统一事件如下：

### 5.1 主流程事件

- START_INTERVIEW
- QUESTION_GENERATED
- QUESTION_PRESENTED
- ANSWER_SUBMITTED
- ANSWER_EVALUATED
- NEXT_QUESTION_READY
- INTERVIEW_COMPLETED
- INTERVIEW_RECOVERED
- INTERVIEW_CANCELLED
- INTERVIEW_FAILED

### 5.2 虚拟人桥接事件

- VH_CONNECT_REQUESTED
- VH_CONNECTED
- VH_CONNECT_FAILED
- VH_SPEECH_STARTED
- VH_SPEECH_FINISHED
- VH_RECORDING_STARTED
- VH_RECORDING_FINISHED
- VH_DEGRADED
- VH_RECOVERED

## 6. 关键状态流转

### 6.1 标准文本面试

```text
idle
→ preparing
→ asking
→ waiting_answer
→ evaluating
→ transitioning
→ asking
→ ...
→ scoring
→ completed
```

### 6.2 虚拟人增强面试

```text
idle
→ preparing
→ asking
→ VH_CONNECTED 时触发播报
→ QUESTION_PRESENTED
→ waiting_answer
→ evaluating
→ transitioning
→ asking
```

虚拟人只影响“问题如何呈现”，不影响“问题如何生成”。

### 6.3 虚拟人失败降级

```text
asking + connected
→ VH_CONNECT_FAILED 或 VH_DEGRADED
→ 保持主状态不变
→ 切换为文本展示问题
→ waiting_answer
```

核心原则：

- 不回滚已生成的问题
- 不丢失已保存的 candidateAnswers
- 不重置 totalQuestions 和 currentQuestionIndex

## 7. 数据流设计

### 7.1 问题生成

1. useInterview 生成问题文本
2. 写入 messages 与 InterviewStorage
3. 如果虚拟人可用，则 useVirtualHumanBridge 调用 speakText
4. 播报完成后派发 QUESTION_PRESENTED
5. 进入 waiting_answer

### 7.2 候选人回答

输入来源允许双通道：

- 文本输入栏
- 虚拟人录音转写

但提交入口必须统一：

- 统一落到 submitAnswer(answerText)

### 7.3 实时评分

评分仍由 useInterview 串行执行：

- submitAnswer
- rateAnswerWithAI
- InterviewStorage.updateAnswerRealTimeScore
- 再进入下一题生成

虚拟人不参与评分。

## 8. 恢复与容错

### 8.1 页面异常

- 由路由级 ErrorBoundary 捕获
- 重新加载后依赖 current_interview_session 恢复

### 8.2 虚拟人连接异常

- 保持面试主状态继续
- UI 上标识“已切换文本模式”
- 不阻塞 submitAnswer 与 completeInterview

### 8.3 浏览器媒体权限失败

- 禁止启用语音回答
- 保留文本输入
- 提示用户改用键盘作答

## 9. 拆分与迁移方案

### 阶段 1：桥接落地

- 保留 useInterview 作为主状态机
- 为 VirtualHumanSDK 增加纯播报和连接状态桥接
- ChatPage 用 bridge 把问题播报接到虚拟人

完成标准：

- 面试首题和后续问题可被虚拟人播报
- 虚拟人断连不影响面试继续

### 阶段 2：剥离旧状态机

- 移除 VirtualHumanSDK 内部 useInterviewState 的面试流程职责
- 保留必要的播放器、录音、连接控制逻辑
- 将 startInterview、handleNextQuestion、askQuestion 等面试流程 API 从 SDK 中下沉或删除

完成标准：

- VirtualHumanSDK 不再保存 currentQuestionIndex
- 题目推进只由 useInterview 驱动

### 阶段 3：引入 useVirtualHumanBridge

- 将 ChatPage 中与数字人相关的副作用抽到独立 hook
- 统一处理播报、录音、回调、降级与提示文案

完成标准：

- ChatPage 不再直接写虚拟人播报副作用
- 虚拟人相关副作用集中在桥接层

### 阶段 4：语音回答闭环

- 虚拟人录音结束后输出转写文本
- 自动填充到 submitAnswer
- 保留人工编辑确认开关

完成标准：

- 候选人可通过语音作答
- 转写失败时可回退到文本手输

## 10. 接口建议

### 10.1 VirtualHumanSDK 对外接口

建议长期收敛为：

```ts
type VirtualHumanHandle = {
  connectAvatar(): Promise<void>
  disconnectAvatar(): Promise<void>
  speakText(text: string): Promise<boolean>
  startRecording(): Promise<void>
  stopRecording(): Promise<void>
  resumePlayback(): Promise<void>
  checkConnectionStatus(): boolean
}
```

建议移除或废弃：

- startInterview
- stopInterview
- askQuestion
- handleNextQuestion
- reAskCurrentQuestion

### 10.2 Bridge Hook 输出

```ts
type UseVirtualHumanBridgeResult = {
  mode: 'text' | 'virtual-human'
  status: 'disconnected' | 'connecting' | 'connected' | 'degraded' | 'error'
  speak(text: string): Promise<void>
  requestAnswerCapture(): Promise<void>
  downgrade(reason: string): void
}
```

## 11. 验收标准

满足以下条件才算完成统一状态机迁移：

- 企业端面试只保留一个问题索引来源
- 题目生成、评分、存储、恢复全部由 useInterview 负责
- VirtualHumanSDK 不再推进题目流程
- 虚拟人断连后，面试能在同一会话中继续完成
- 至少具备页面异常恢复、数字人挂载、问题播报三类测试

## 12. 当前建议的下一实施顺序

建议按以下顺序推进：

1. 提取 useVirtualHumanBridge，收敛 ChatPage 内的 speakWithVirtualHuman 与状态处理。
2. 将 VirtualHumanSDK 中面试相关 API 标为 deprecated，并移除对 useInterviewState 的流程依赖。
3. 为“播报完成后再进入 waiting_answer”补状态事件，避免问题还未读完就允许作答。
4. 设计语音转写到 submitAnswer 的确认交互。
