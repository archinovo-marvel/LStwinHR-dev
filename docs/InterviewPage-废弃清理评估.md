# InterviewPage 废弃清理评估

> 评估日期：2026-05-08
> 目标文件：client/src/pages/InterviewPage.js

## 1. 当前结论

InterviewPage 已处于“逻辑废弃但代码仍保留”的状态。

当前判断依据：

- 文件头已经标记 @deprecated。
- App 路由中已不存在企业端 InterviewPage 的入口。
- 企业端实际面试主链路已经迁移到 ChatPage + useInterview。

因此：

- InterviewPage 不再属于线上主流程。
- 继续保留的价值只剩“历史参考”。
- 继续保留的风险大于价值。

## 2. 当前残余风险

### 2.1 残留错误实现仍然存在

InterviewPage 内仍保留以下旧问题：

- 使用 temp-candidate-id 作为候选人 ID。
- 仍依赖旧的评估与提交方式。
- 与当前企业端主链路的评分、存储、恢复策略不一致。

虽然这些问题目前不影响线上路由，但会带来两个隐患：

- 新成员误以为该文件仍可复用。
- 后续搜索相同功能时，容易被过期实现误导。

## 3. 保留价值评估

目前保留价值较低。

可以保留的唯一理由：

- 用作历史演进参考，帮助对比旧版固定题面试与新版 useInterview 主链路的差异。

但这个价值更适合通过文档保留，而不是继续保留可执行页面代码。

## 4. 建议的删除条件

满足以下条件后，建议直接删除 InterviewPage.js：

1. 企业端主链路 ChatPage + useInterview 已覆盖旧页面需要的核心能力。
2. 团队不再需要查看旧版页面 UI 实现细节。
3. 已有文档记录旧版与新版差异。
4. 代码搜索中不再存在对 InterviewPage 的直接引用或隐式依赖。

基于当前代码状态，这 4 个条件里至少前 1、3、4 已基本满足。

## 5. 建议的清理路径

### 方案 A：立即删除

适用条件：

- 团队确认不再参考旧页面。

动作：

- 删除 client/src/pages/InterviewPage.js

优点：

- 彻底消除误导。
- 清掉 temp-candidate-id 残留实现。

风险：

- 若有人仍希望查看旧版页面细节，需要改从文档或 git 历史获取。

### 方案 B：先冻结后删

适用条件：

- 团队还想保留一个迭代周期的观察期。

动作：

- 保留文件，但重命名为 InterviewPage.deprecated.js 或移入 archive 目录。
- 在文件顶部增加“禁止引用”的说明。

优点：

- 降低误用概率。

风险：

- 仍保留一份过时代码。

## 6. 当前推荐

当前更推荐方案 A：直接删除。

原因：

- 路由已经移除，删除不会影响主流程。
- 该文件仍保留 temp-candidate-id 等明显错误实现。
- 当前 docs 已经开始承接架构与优化记录，历史参考更适合留在文档或 git 历史中。

## 7. 若下一步执行清理

建议顺序：

1. 删除 InterviewPage.js。
2. 全局搜索并确认无引用残留。
3. 在优化记录文档中标记“旧版企业端 InterviewPage 已删除”。
